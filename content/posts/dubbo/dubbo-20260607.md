# dubbo-go 应用级元数据定时续约与 GC 清理

> 对应 Issue: [#3355](https://github.com/apache/dubbo-go/issues/3355)
> 分支: `feat-3355-metadata-renew-and-gc`

## 背景与问题

Dubbo 应用级服务发现中，元数据模式为 remote 时，Provider 启动会向注册中心注册一个 `/dubbo/{application}/{revision}` 节点，其中 `revision` 由接口列表计算得出。当接口发生变动（如新增、删除方法），会生成一个新的 revision 并注册新节点，但老的 revision 节点不会被自动删除。版本发布多次后，元数据中会累积大量无用的旧 revision 配置，造成资源浪费和查询性能下降。

## 讨论达成的方案

Issue 评论区经过讨论，最终达成以下共识：

1. **定时心跳上报**：remote 模式下应用元数据每天定时上报（续约），刷新 `lastUpdatedTime`。
2. **过期检测与清理**：上报完成后检测其他 revision 是否超过 N 天未更新，再交叉验证当前存活实例中是否已无该 revision，无则删除。

## 详细改动

### MetadataReport 接口扩展

**文件**: `metadata/report/report.go`

新增三个接口方法：

```go
// AppRevision represents a revision entry with its last modification time.
type AppRevision struct {
    Revision   string
    ModifyTime int64 // unix timestamp in milliseconds
}

// 解析元数据 JSON 中的 lastUpdatedTime 字段
func ParseMetadataLastUpdatedTime(data []byte) int64

// 新增接口方法
UnPublishAppMetadata(application, revision string) error  // 幂等删除
ListAppRevisions(application string) ([]AppRevision, error) // 列出所有 revision
URL() *common.URL  // 返回创建该 report 的 URL（用于读取配置参数）
```

**设计决策**：`ModifyTime` 不依赖 Zookeeper/Etcd 的节点 stat（ctime/mtime），而是统一从 JSON 内容中的 `lastUpdatedTime` 字段解析。这保证跨后端语义一致，也使 Nacos（基于 config API）能统一处理。

### MetadataInfo 数据模型

**文件**: `metadata/info/metadata_info.go`

新增字段和方法：

```go
type MetadataInfo struct {
    // ... existing fields
    LastUpdatedTime int64 `json:"lastUpdatedTime,omitempty" hessian:"-"`
}

// Snapshot creates a deep copy for safe concurrent access
func (info *MetadataInfo) Snapshot() MetadataInfo
```

**设计决策**：`Snapshot()` 做深拷贝是因为续约操作在定时器 goroutine 中执行，需避免与主协程的并发修改冲突。`hessian:"-"` 标签确保该字段不参与 Hessian 序列化（仅用于 JSON/远程存储）。

### 定时续约 (Renew)

**文件**: `registry/servicediscovery/service_discovery_registry.go`

#### 调度机制

```
startRenewAppMetadataTimer()
    │
    ├─ 启动时立即执行一次（可配置 metadata.renew-on-startup=false 关闭）
    │
    └─ 注册 time.AfterFunc，首次触发时间 = 次日凌晨 2:00 + [0, 4h) 随机偏移
       触发后每 24h 循环执行
```

**随机偏移**：0~4 小时随机抖动，防止同一应用多个实例同时续约造成惊群。

#### 续约逻辑 (`doRenewAppMetadata`)

```
1. 获取当前 registryMetadataInfo 的 Snapshot
2. 设置 LastUpdatedTime = now.UnixMilli()
3. 调用 PublishAppMetadata 覆盖写（刷新元数据中的时间戳）
4. 若 gc.enabled=true，执行 doGarbageCollect()
```

#### 前置条件

- `metadataReport != nil`
- `reportURL.GetParamBool("cycle.report", true)` 为 true
- `metaInfo.Revision != "0"`（空 revision 无需续约）

### GC 清理 (`doGarbageCollect`)

**文件**: `registry/servicediscovery/service_discovery_registry.go`

四步流程：

```
Step 1: ListAppRevisions(app)
    ↓ 获取该应用在元数据中心的所有 revision 列表
Step 2: 筛选过期候选
    ↓ 过滤条件：
    │  - 跳过特殊 revision: "0", "N/A", "", 当前实例 revision
    │  - ModifyTime == 0 → 跳过（老版本未设置时间戳，无法判断过期）
    │  - ModifyTime < cutoff (now - gc.window 天) → 标记为候选
Step 3: 获取存活实例
    ↓ serviceDiscovery.GetInstances(app)
    │  提取每个实例 metadata 中的 "dubbo.exported-services.revisions"
    │  构建 aliveRevisions map
Step 4: 安全删除
    ↓ 对每个候选 revision：
       若不在 aliveRevisions 中 → 调用 UnPublishAppMetadata 删除
```

**安全保证**：
- 仅删除**既过期又无存活实例引用**的 revision
- `ModifyTime == 0` 的条目不会被误删（兼容老版本）
- 当前实例自身的 revision 始终被跳过
- 窗口期默认 5 天，极端场景（N 天未上报恰好检测时启动）风险极低

### 配置项

**文件**: `common/constant/key.go`

| 配置键 | 默认值 | 说明 |
|--------|--------|------|
| `metadata.gc.enabled` | `true` | 是否启用 GC 清理 |
| `metadata.gc.window` | `5` (天) | GC 过期窗口，超过此天数未更新的 revision 成为候选 |
| `metadata.renew-on-startup` | `true` | 启动时是否立即执行一次续约 |
| `cycle.report` (已有) | `true` | 是否启用定时续约（同时控制 Timer 是否启动） |

以上参数通过 metadata report URL 配置。

```go
type serviceDiscoveryRegistry struct {
    // ... existing fields

```

Timer 在 `Destroy` 时被 Stop 并置 nil，防止销毁后触发。

## 已知风险

1. **GC 竞态风险**：若持有某 revision 的服务下线超过 N 天，被新服务判定为过期候选；恰在此时老服务重新上线并写入元数据，此时新服务将该刚写入的元数据删除，导致老服务处于无元数据状态。正常流程下该场景发生的概率极低.
