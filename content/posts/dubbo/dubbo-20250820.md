## 背景

A调用B, A进行了异常捕捉, 如果是自定义的业务异常, 那么就会直接用这个异常的错误信息返回给前端, 如果不是, 那么就会用兜底的未知异常信息返回.

发现B明明抛出的是业务异常, 但是前端收到的却是未知异常, 导致很不友好

## ExceptionFilter

debug了半天, 最终发现是dubbo有一个`org.apache.dubbo.rpc.filter.ExceptionFilter`在provider上生效, 对Runtimexception且没有在接口上声明的, 会转换成字符并重新new一个Runtimexception, 目的是为了减少消费端反序列化失败的可能. 就是这里把业务异常转换成了Runtimexception, 最终导致A无法处理, 只能给未知异常信息.

```java
@Override
public void onResponse(Result appResponse, Invoker<?> invoker, Invocation invocation) {
    if (appResponse.hasException() && GenericService.class != invoker.getInterface()) {
        try {
            Throwable exception = appResponse.getException();

            // 受检异常
            if (!(exception instanceof RuntimeException) && (exception instanceof Exception)) {
                return;
            }
            // 在方法上进行了声明
            try {
                Method method = invoker.getInterface().getMethod(invocation.getMethodName(), invocation.getParameterTypes());
                Class<?>[] exceptionClasses = method.getExceptionTypes();
                for (Class<?> exceptionClass : exceptionClasses) {
                    if (exception.getClass().equals(exceptionClass)) {
                        return;
                    }
                }
            } catch (NoSuchMethodException e) {
                return;
            }

            logger.error(CONFIG_FILTER_VALIDATION_EXCEPTION, "", "", "Got unchecked and undeclared exception which called by " + RpcContext.getServiceContext().getRemoteHost() + ". service: " + invoker.getInterface().getName() + ", method: " + invocation.getMethodName() + ", exception: " + exception.getClass().getName() + ": " + exception.getMessage(), exception);

            // 异常和接口在同一个包内
            String serviceFile = ReflectUtils.getCodeBase(invoker.getInterface());
            String exceptionFile = ReflectUtils.getCodeBase(exception.getClass());
            if (serviceFile == null || exceptionFile == null || serviceFile.equals(exceptionFile)) {
                return;
            }
            // jdk的异常
            String className = exception.getClass().getName();
            if (className.startsWith("java.") || className.startsWith("javax.")) {
                return;
            }
            // dubbo rpc异常
            if (exception instanceof RpcException) {
                return;
            }

            // 就是这里, 把异常转成字符串并重新new了RuntimeException抛出
            appResponse.setException(new RuntimeException(StringUtils.toString(exception)));
        } catch (Throwable e) {
            logger.warn(CONFIG_FILTER_VALIDATION_EXCEPTION, "", "", "Fail to ExceptionFilter when called by " + RpcContext.getServiceContext().getRemoteHost() + ". service: " + invoker.getInterface().getName() + ", method: " + invocation.getMethodName() + ", exception: " + e.getClass().getName() + ": " + e.getMessage(), e);
        }
    }
}
```



## 修复

因为业务异常是定义在公共包里的, 所以在接口上进行了声明, 果然成功抛出.

发现在其他项目里, 提供方会自定义一个ExceptionFilter
