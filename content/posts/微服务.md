仅限于对微服务以及一些组件的简单介绍.

## 微服务
微服务的概念是相较于单体应用而来的, 在解释什么是微服务和为什么需要微服务时, 就首先需要了解什么是单体应用, 以及单体应用遇到的问题.
## 单体应用
所有功能模块部署在一个单独的应用中, 编译成一个非常大的包启动. 单个节点承载不了压力的时候, 也可以部署多节点, 需在前面部署一个负载均衡器
![](/微服务/单体.png)

缺点
- 开发效率低：所有的开发在一个项目改代码，递交代码相互等待，代码冲突不断
- 部署不灵活：构建时间长，启动慢, 任何小修改必须重新构建整个项目
- 稳定性不高：一个微不足道的小问题，可以导致整个应用挂掉
- 资源利用低:  单体应用扩展性能时只能整个应用进行扩展
优点
- 架构简单, 项目初创时可以快速上线, 没有微服务的心智负担
- 服务调用没有网络消耗, 适合游戏主程等时延敏感的场景

## 微服务
以技术或者业务等标准对各个功能模块拆分. 各个服务单独部署和发布, 可以按需扩容, 服务之间通过网络通信
![](/微服务/微服务.png)

拆分标准

- 技术: 算法服务技术栈不同, 往往使用serverless函数服务提供调用
- 业务: 用户中心, 订单中心, 商品中心, 权限中心...架构师需要根据自己的经验做出合理的拆分

优点
- 架构上确定了边界, 业务开发可以专注本模块的开发, 发挥自己的业务经验
- 只需遵守通信协议, 可以自由选择技术栈
- 隔离性, 一个模块挂掉了, 不影响其他功能的使用
- 容易扩展, 可以针对性的扩容以应对高并发的压力
缺点
- 心智负担高: 分布式一致性, 分布式事务
- 服务治理困难: 服务注册和发现, 负载均衡, 容错, 限流, 监控告警, 链路追踪, 日志采集, 服务编排
- devops成本高: 项目拆分后, 分支, 打包, 发布需要自动化管理

可以看到, 微服务不是银弹, 并不试用于所有场景, 即便在它试用的领域, 也往往只有在问题规模到一定程度之后, 才能从它的优势中获益.

## rpc
微服务首先需要解决网络通信问题, 这里介绍一下我了解的几个rpc框架.

### grpc
由 Google 开发，是一款语言中立的, 基于http2协议, header和payload分离, 使用protobuf序列化的rpc框架. 其在req-resp模型上还提供了流式模型. 在grpc-java项目的examples目录里面有非常多的例子, 这里就不贴了. 但是应该注意到, 在这些例子中, 地址都是硬编码在代码中的, 确实grpc没有提供服务注册发现, 负载均衡等功能(官方推荐的方案是使用dns等), 而我们知道这些能力对于微服务来说是不可或缺的, 这是否意味着相较于dubbo这样功能完备的rpc框架, grpc不可用呢? 其实不是的, grpc在设计上更专注于通信层, 我想在google设计这个rpc框架的时候应该就是结合了servicemesh的云原生场景. 如果仅仅只是想使用流式模型等grpc功能而并不想直接上servicemesh, 也可以把grpc托管到dubbo这样完备的框架上, 详见:dubbo-grpc协议

### 流式模型
相较于目前使用的dubbo2, 调用一次拿到一个返回值的req-resp模型, grpc提供的流式模型可能是一个不太熟悉的地方, grpc的流式模型分为客户端流式, 服务端流式和双向流式, 说白了就是可以把一个数据集合拆分成多次发送对方, 这里我想了几个例子来介绍这种模型的优势
1. 发送一个大数据包: 一个超大的数据包整体序列化时, 容易造成oom, 也容易把cpu占满, 分多次传输可以把压力打平, 保证应用稳定
2. 语音,视频流等流式场景: 在这些场景中, 天然需要流式模型
3. limit 10000,10: 分布式数据库多节点排序, 如果是req-resp模型, 那么就需要把每个节点的前10010行缓存到主节点进行整体的一个排序, 流式模型或许可能每个节点都只传输前几千行就可以得到结果. 一些分库分表中间件就会做这样的优化.

### dubbo
![](/微服务/dubbo.png)
dubbo框架是一个开箱即用的高性能rpc框架, 目前主要分为dubbo2和dubbo3两个大版本. dubbo3为了适应云原生, 服务发现修改成了应用级, 主推通信协议从dubbo修改为了Triple, Triple协议可以认为是在grpc的协议上添加了一些header, 由此也可以看到grpc的一个扩展能力.

#### dubbo2协议
![](/微服务/dubbo2.png)

关于协议细节可以查看官方文档:上图是使用wireshark对json序列化时抓的包, 可以看到上下文信息是放在最后面的, 如果想使用servicemesh的路由功能, 那么就必须解析整个包, 这也是dubbo2不适用于云原生的一个原因.

### 服务治理
dubbo自带了对各种注册中心的支持, 常用的有zookeeper和nacos.实现了随机, 加权轮询, 一致性 Hash等负载均衡算法服务降级容错, 有快速失败, 重试等

## springcloud
springcloud是全家桶式的一个技术栈, 各种组件非常多.注册中心Eureka, 客户端Feign, 负载均衡Ribbon, 限流熔断Hystrix, 网关Zuul, 配置中心git等等, 这些组件也都是可以替换的, 列举的是最经典的实现而已. 默认使用http作为通信协议, 在高并发场景下性能较低, 不过其实大部分公司也触碰不到这个性能天花板就是了. 如果限于java技术栈的话, 这是最适合开箱即用的rpc框架.

### docker
在介绍k8s前, 肯定要先对k8s中运行的容器先进行介绍, 一般而言也就是docker.其实k8s支持的是容器化标准, 并不仅限于docker, 例如还支持podman, 不过我只对docker了解一点. docker可以提供一个隔离的虚拟环境, 但又不同于virtualbox等虚拟机, 其占用的资源非常少, 可以用来保证测试和线上环境的一致.

#### 镜像
镜像是创建Docker容器的基础, 可以认为就是一个文件, 可以在网络上传输分享. 镜像是只读的, 在一个镜像的基础上创建一个新的镜像, 只是在原有的文件上创建了一层增量. 这可以减少磁盘占用. docker pull 下载镜像, 在命令中指定仓库地址,镜像名称和tag. 

docker images 可以查看本地的镜像列表. 

docker rmi 删除本地镜像文件.

docker history 查看镜像构建命令, 可以从中学习如何构建自己的镜像

docker inspect 查看镜像内部信息, 申明的端口,挂载等

#### 容器
容器是镜像运行的实例, 其中会有运行中的进程, 往往一个容器内也只有一个进程. 容器在启动的镜像上创建了一层可写的文件层

docker run 启动一个容器并运行

docker ps 查看运行中的容器, 加上 -a 可以查看所有容器, 包括失败或者停止运行的

docker rm 删除容器

docker exec -it xxxx /bin/bash 进入容器

docker inspect xxx 查看容器信息

docker cp 在宿主机和容器间复制文件

docker update 修改一些参数

docker logs 查看日志

```shell
# mysql容器, docker的网络io性能不错, 但文件io目前尚有性能问题, 其实不适于数据库
# --name指定容器名字
# -v 挂载, 容器内的文件不会被持久化, 配置文件, 数据文件, 和日志文件最好挂载到宿主机
# -d 守护运行
# -p 端口映射, 容器的端口外部不能访问, 容器之间可以通过link关联
# -e 环境变量
# --restart=always 容器策略为一直
docker run --name mysql \
-v /Users/guzemin/docker/mysql/data:/var/lib/mysql \
-v /Users/guzemin/docker/mysql/conf/my.cnf:/etc/mysql/my.cnf \
-v /Users/guzemin/docker/localtime:/etc/localtime \
--restart=always \
-d -p3306:3306 -e MYSQL_ROOT_PASSWORD=123456 -e TZ=Asia/Shanghai mysql/mysql-server:8.0.25
```

#### master
管理控制中枢，所有任务分配都来自于此。Master节点主要由四个模块组成：API Server、scheduler、controllermanager、etcd。master可以作为node运行pod, 也可以配置不运行
api server: 提供了Kubernetes资源对象的唯一操作入口
conroller manager: 集群内部的管理控制中心, 管理pod, service等
Etcd: 存储状态, 配置等
scheduler: 主要作用是将待调度的Pod按照给定的调度算法和调度策略绑定到Kubernetes集群中的某个合适的Node上
#### Namespace
命名空间, 在逻辑上对资源进行隔离
#### pod
k8s调度的基础单元, 可以将它看做一个虚拟的服务器. 一个pod可以包含多个容器, 大部分时候一个pod只会有一个容器, 但是例如部署了servicemesh的话, 会在每个pod中同时启动一个sidecar, 那样一个pod中就有两个容器了. 多个相同的pod可能运行在一个node节点上.
pod内的容器共享网络和磁盘等资源. pod中的容器需要一直被运行, 否则停止后pod就会删除, 此时ReplicaSet因为节点数量不满足要求, 就会又重新启动一个pod.
每个pod都会有一个集群内的id, 这个ip可以被其他pod访问到, pod内部的容器间可以直接通过localhost访问.

kubectl get pods --namespace=default 查看默认命名空间的pods

kubectl describe pod xxxx 查看某个pod信息

kubectl logs xxx 查看pod日志

kubectl exec -it xxx -- /bin/bash 进入pod

kubectl cp 复制文件

#### ReplicaSet
望文生义可以得知, 负责控制pod的副本数量, 用来管理pod的缩容和扩容, 但往往不单独使用, 而是通过deployment被使用. rs用来保证pod的数量满足一定的要求, 如果发现pod的数量小于要求, 那么就会启动调度在某个节点上把pod启动, 如果pod数量大于要求, 它就会关闭多余的节点.

kubectl get rs --n=default

#### deployment
为Pod和ReplicaSet提供了一个声明式定义(declarative)方法, 通常不会直接创建pod和rs, 而是通过deployment来配置. 简单来说就是对应微服务中的一个服务. 无状态应用.
![](/微服务/deployment.png)
![](/微服务/deployment2.png)

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  # 部署名字
  name: test-k8s
spec:
  replicas: 2
  # 滚动更新
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0
  # 用来查找关联的 Pod，所有标签都匹配才行
  selector:
    matchLabels:
      app: test-k8s
  # 定义 Pod 相关数据
  template:
    metadata:
      labels:
      # 注意标签必须和selector匹配, 否则将生成该pod
        app: test-k8s
    spec:
      # 定义容器，可以多个
      containers:
      - name: test-k8s # 容器名字
        image: ccr.ccs.tencentyun.com/k8s-tutorial/test-k8s:v1 # 镜像
        # 就绪探针
        readinessProbe:
                tcpSocket:
                port: 8080
                initialDelaySeconds: 5
                periodSeconds: 10
          # 存活探针
                livenessProbe:
                tcpSocket:
                port: 8080
                initialDelaySeconds: 15
                periodSeconds: 20
        # 资源限制, 分cpu和内存, requests是最初申请的资源, limit是最大可以申请资源
        resources:
          limits:
            cpu: 500m
          requests:
            cpu: 200m
            memory: 256Mi
```



kubectrl create -f test-k8s.yaml

kubectl scale deployment test-k8s --replicas=3 扩容缩容

kubectl set image -n default deployment/test-k8s test-k8s=ccr.ccs.tencentyun.com/k8s-tutorial/test-k8s:v2 更新镜像

kubectl rollout history deployment test-k8s 查看版本历史

kubectl rollout undo deployment test-k8s --to-revision=2 回滚到指定版本

#### hpa
deployment可以手动扩容缩容, 但是大部分时候应用的负载是突然升高的, 这时候人不可能第一时间介入, 很容易导致服务崩溃, 甚至链式导致其他服务崩溃, 这时候就需要有一个可以自动处理的机制, 就是hpa, 弹性伸缩.
hpa可以根据cpu或,内存或者每秒请求数等指标, 来判断是否需要扩容和缩容. 在负载上升时自动扩容, 在负载下降时自动缩容
```yaml
apiVersion: autoscaling/v2beta2
kind: HorizontalPodAutoscaler
metadata:
  name: php-apache
  namespace: default
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: php-apache
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 50
```

#### 滚动更新
微服务发布频繁, 在发布时当然需要做到平滑的应用上下线, 降低线上影响. k8s提供了滚动更新的能力, 在更新deployment时, 可以启动一部分新pod, 启动完成后再关闭一部分旧pod, 再启动一部分新pod, 再关闭, 直至全部pod都更新完成, 这将保证任何时候都存在足够数量的pod提供服务.
spec.strategy.type.rollingUpdate.maxSurge 最大可以创建多少个pod。这个值可以是一个绝对值数字，也可以是个百分比
spec.strategy.type.rollingUpdate.maxUnavailable 最大不可用的pods的值。可以是一个绝对值数字，也可以是个百分比。

#### 探针
在滚动更新时, 其实有个问题没有解决, 什么时候判断新pod已经启动了, 仅仅是进程是否被拉起么?这显然是不行的, 因为我们知道进程启动到应用可以提供服务, 往往还有一个很漫长的过程, 如果这个时候就把旧的pod关闭了, 那么线上就是一堆无法提供服务的pod. 
另一个问题就是如何判断pod是否存活, 可能pod内的进程没有挂, 但是由于某些原因该pod已无法提供服务了, 那么虽然rs保证了pod的数量, 但是仍旧会出现线上故障.
k8s有多种探针在pod的各个生命周期阶段来保障k8s可以正确判断pod的状态. 分为三种探测方法: exec, tcp, http, 每个应用可以按照自己的实际情况进行配置. 

#### DaemonSet
在每个node节点上跑一个 Pod，可以用来做节点监控、节点日志收集等. 比方说filebeat, 所有应用日志挂载出来到node的指定目录上, 一个filebeat进程就能处理, 节约资源, 那么它的pod数量就是和node的数量是一一对应的. 或者再比如服务器监控, 肯定也是每个服务器一个节点.

#### StatefulSet
相较于deployment, StatefulSet是有状态的服务, 类似zookeeper集群, 每个节点需要有自己的唯一id, 且部署后该id不应被改变, 如果id为1的节点挂了, 那么就需要k8s正确的拉起id为1的节点, 否则整个zookeeper集群可能就会无法使用.

### 链路追踪 tracing
我们为什么需要链路跟踪? 在微服务架构下, 服务之间的调用关系往往变得非常复杂, 即便是系统的架构师在缺乏合适的工具的情况下, 也无法理清. 而一次请求如果慢了或者出了问题, 也无法快速定位是哪个服务引起的. 也就是在调用链路的复杂性超过人能控制的范围时就需要链路追踪这样的工具来协助服务治理.
链路跟踪的很多概念都来源于google一篇关于Dapper的论文, 其中关键是trace和span两个概念.trace代表全局唯一的一次请求, 作为一个上下文在调用间传递, 所以显然需要rpc框架支持.span代表一个时间区间, 有自己的id和一个parentId. span中通过annotation存储需要的数据, 一般会有以下四个annotation

cs - Clent Sent 客户端发送了请求

sr - Server Receive 服务端接受到请求

ss - Server Send 服务端处理完毕，向客户端发送回应

cr - Client Receive 客户端收到结果

#### pinpoint
使用javaagent, 对代码无侵入. UI相较于zipkin更丰富, 有更详细的span信息, 已支持dubbo插件, 有类信息, 对常用数据库甚至可以查看sql语句, 和redis命令. 在trace之外还支持了metrics
主要组件如下:

HBase: 存储数据, 注意不支持2.x版本

Pinpoint Collector: 负责收集agent上传的消息

Pinpoint Web: web界面

Pinpoint Agent: java启动时需配置

## 分布式事务
### 事务
有四个隔离级别, 读未提交/读已提交/可重复读/序列化

读未提交会出现脏读的问题, 读取另一个事务未提交的数据

读已提交会出现不可重复读, 两次读取数据内容不一致, 因为两次读取之间有新事务修改数据

可重复读会出现幻读, 两次读取数据条数不一致

### mvcc
读不加锁，读写不冲突。在读多写少的OLTP应用中，读写不冲突是非常重要的，极大的增加了系统的并发性能. mvcc只在读已提交和可重复读下工作, 这里介绍innodb的实现机制

![](/微服务/mvcc.png)
![](/微服务/mvcc2.png)

在查询时会开启一个read view,把当前未提交的事务id构成一个数组缓存起来,然后通过主键id找到要查询的数据行的最新的事务id.
1. 如果要查询的数据行的事务id小于read view中最小的事务id,说明该行数据事务在查询前已提交,可被查看
2. 如果要查询的数据行的事务id大于read view中最大的事务id,说明该行数据事务在查询后提交,不可被查看,通过其回滚指针查看上一次提交的事务id,然后重新从1开始判断
3. 如果要查询的数据行的事务id在read view的事务id区间内,判断该事务id是否在列表中存在,如果存在则说明此数据还没有提交,因此不可见,通过其回滚指针查看上一次提交的事务id,然后重新从1开始判断.如果不存在,说明该数据的事务已提交,那么读已提交显然可见,可重复读仍不可见,通过其回滚指针查看上一次提交的事务id,然后重新从1开始判断在

innodb中(默认repeatable read级别)，事务在begin/start transaction之后的第一条select读操作后，会创建一个快照(Read View)，将当前系统中活跃的其他事务记录记录起来

在innodb中(read committed级别)，事务中每条select语句都会创建一个快照(Read View)

这里也介绍一下redolog, 数据修改后, 为了持久化, 显然需要把改动写入磁盘, 但是如果直接写innodb的数据扇, 会存在大量的随机读写, 导致性能下降, 所以innodb是先把日志写入redolog中, redolog是一个循环缓存, 是顺序写入, 后续会异步的把改动刷新到数据扇中. mysql挂掉恢复时会通过redolog和binlog保证提交事务不丢失.

### acid/cap/base
#### 2pc/xa
![](/微服务/2pc.png)

准备阶段（Prepare phase）：事务管理器给每个参与者发送 Prepare 消息，每个数据库参与者在本地执行事务，并写本地的 Undo/Redo 日志，此时事务没有提交。（Undo 日志是记录修改前的数据，用于数据库回滚，Redo 日志是记录修改后的数据，用于提交事务后写入数据文件）

提交阶段（commit phase）：如果事务管理器收到了参与者的执行失败或者超时消息时，直接给每个参与者发送回滚（Rollback）消息；否则，发送提交（Commit）消息；参与者根据事务管理器的指令执行提交或者回滚操作，并释放事务处理过程中使用的锁资源。注意：必须在最后阶段释放锁资源。

性能问题。从流程上我们可以看得出，其最大缺点就在于它的执行过程中间，节点都处于阻塞状态。各个操作数据库的节点此时都占用着数据库资源，只有当所有节点准备完毕，事务协调者才会通知进行全局提交，参与者进行本地事务提交后才会释放资源。这样的过程会比较漫长，对性能影响比较大。

协调者单点故障问题。事务协调者是整个XA模型的核心，一旦事务协调者节点挂掉，会导致参与者收不到提交或回滚的通知，从而导致参与者节点始终处于事务无法完成的中间状态。
丢失消息导致的数据不一致问题。在第二个阶段，如果发生局部网络问题，一部分事务参与者收到了提交消息，另一部分事务参与者没收到提交消息，那么就会导致节点间数据的不一致问题。

2pc在mysql中还存在另一个问题, 即数据库的主备切换, undo/redo日志是innodb引擎级别的日志, 不是数据库级别的日志, 数据库级别的同步是通过binlog日志同步的, 所以主从节点间不会同步undo/redo日志, pre阶段完成后出现主备切换后备库没有undo/redo日志, 无法commit.

2pc在极端场景还是会出现不一致

#### 3pc
相较于2pc在前面新增了一个CanCommit阶段, 然后参与者也加了一个超时机制, 超时会自动commit. 因为经过cancommit的状态同步, 所以大概率超时自动commit是没问题的, 当然存在不一致的风险.

#### tcc
Try阶段：主要是对业务系统做检测及资源预留。

Confirm阶段：确认执行业务操作。

Cancel阶段：取消执行业务操作。

最经典的场景就是通过表中新增一个预扣款字段, 在try阶段扣除金额到这个预扣款字段中, 如果事务成功就把预扣款扣除, 失败就把预扣款再加回到金额中. 可以看到tcc是业务层面的xa

#### 幂等
try, Confirm和cancel接口必须保证幂等.

除了幂等, tcc也需要Confirm和cancel这两个接口是必然能成功的. 否则也会出现数据不一致

#### 空回滚
在没有调用Try方法的情况下，调用了二阶段的Cancel方法. 产生原因是事务参与者在try时出现了网络异常等情况, 协调者发现try没有成功, 就调用各参与者的cancel方法, 但是这个时候刚才有问题的参与者又恢复正常被调用了cancel.

#### 悬挂
二阶段Cancel接口比Try接口先执行, rpc通信不能保证顺序.

在tcc的实现框架上, seata实现的比较简单, 仅保证了调用了三个接口, 对于上面的几个问题需自行保证, 类似Hmily等tcc框架通过在本地新增一个记录表等方式在框架层面对这些问题进行了解决.

### seata
#### at
![](/微服务/at.png)

这是seata提出的一个新的分布式事务解决方案, 简单理解可以认为就是xa, 是seata托管的xa. 因为是xa所以数据库必须有事务保证, 另一个新的限制是必须是jdbc

一阶段：业务数据和回滚日志记录在同一个本地事务中提交，释放本地锁和连接资源。

二阶段：提交异步化，非常快速地完成。回滚通过一阶段的回滚日志进行反向补偿。

at模式需要在所有参数者本地数据库中增加一张undo_log表, 用来代替数据库级别的undolog. 一阶段中是如何写入undo_log的呢, seata通过jdbc获取sql然后解析sql, 再根据解析结果查询拿到当前数据的一份快照作为备份.例如: update product set name = 'GTS' where name = 'TXC'; 解析成 select id, name, since from product where name = 'TXC'; 把查询结果序列化放入undo_log表中, 如果最终事务提交成功, 就删除备份, 如果失败就通过这个备份进行回滚. 注意回滚时会进行数据的对比校验, 如果其他非at事务的修改导致了数据不一致可能会出现问题

这个模式的并发性能, 显然高于2pc, 但小于tcc, 但是又不像tcc需要接口和数据库上的修改. 在性能影响上一个是获取全局锁的交互, 一个是备份的几次数据库交互. 主要问题是因为回滚时需要保证数据一致, 那就必须要将所有的改动都托管给at

#### saga
![](/微服务/saga.png)

像2pc, tcc等模式, 都是所有节点状态同步推进, 直至到最终状态, saga不是, 它是每个节点依次推进状态. saga是长事务的一个解决方案, 事务正常提交后继续执行下个事务, 出现异常回滚时, 按照逆向一个个回滚, 回滚类似tcc需要提供相应接口. 它的状态机非常复杂, 我也不是很了解

### 可靠消息最终一致性
通过消息系统保证最终一致性. 例如转账场景, 首先扣款, 然后发送mq消息(事务), 通过mq的ack机制确保加款消息被消费成功. 当然消费需要有幂等保障. 在业务可以接受一定时延的情况下, 该模式是最合理的方案. 我们知道现实中转账就是会有一定延迟的, 有些商城系统付款后订单也不是立即生成的, 甚至后续会出现下单失败退款的情况, 都说明了最终一致性的应用.

### 感想
分布式事务可以看作是分布式一致性的一个场景, 可以看到虽然有各种方案, 但每种方案都带有一定的缺点, 落地也非常复杂, 选型时必须充分考虑业务场景, 没有什么完美的解决方案. 

我的建议是非核心场景不要上分布式事务, 分布式事务对接口性能影响还是较大的, 出现问题也比较难排查,可以通过数据对账等方式手动解决, 在业务设计上也可以解决很多问题, 例如转账先扣款, 出问题可以补款, 或者页面上用户手工重试就能解决. 核心链路非高并发场景, 可以使用xa, 对代码和数据库改动较小, 协调者故障数据不一致的概率很小, 如果能保证at的使用场景, 那么也可以使用at. 不是强一致性要求的话就使用最终一致性. 真的特别高并发的场景, 那么可以考虑tcc.
