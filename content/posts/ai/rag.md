## RAG

### 查询转换策略

朴素 RAG 的典型流程是将文档分块、嵌入，然后检索与用户问题语义相似度高的文档块。但这存在几个问题：文档块可能包含无关内容、用户问题的措辞可能不适合检索、可能需要从用户问题生成结构化查询。

查询转换的核心思想是：**在将用户问题传递给嵌入模型之前，利用 LLM 对其进行变换操作**，以改善检索效果。以下几种方法的共同点是使用 LLM 生成新的（或多个）查询，主要区别在于生成时使用的提示词不同。

> 参考：[LangChain - Query Transformations](https://www.langchain.com/blog/query-transformations)

#### Rewrite-Retrieve-Read（重写-检索-阅读）

利用 LLM **重写用户查询**，而不是直接使用原始查询进行检索。流程变为：重写 → 检索 → 阅读。

原始查询并不总是最适合检索的，尤其在真实场景中，用户的提问往往含糊或措辞不当。通过 LLM 重写为更清晰、更适合检索的版本后再执行检索。

```
原始问题: "那个怎么用"
重写后: "LangChain 中 Multi Vector Retriever 如何使用"
```

#### Step Back Prompting（后退提示）

利用 LLM 生成一个更抽象、更高层次的"后退"问题，然后**同时使用后退问题和原始问题进行检索**，将两个检索结果都用于支撑 LLM 的响应。

后退问题可以帮助获取更广泛的背景知识，弥补原始问题过于具体而检索不到相关文档的不足。

```
原始问题: "Python 3.11 有哪些新特性"
后退问题: "Python 版本更新的历史和新特性概述"
→ 同时用两个问题检索，合并结果
```

#### Follow Up Questions（后续问题处理/上下文压缩）

在对话链中处理后续问题时，存在三种选项：

1. **仅嵌入后续问题** — 会丢失上下文。例如先问"意大利有什么好玩的"，再问"那里有什么食物"，仅嵌入后者将丢失"那里"指代意大利的上下文
2. **嵌入整个对话** — 后续问题若与之前无关，会引入不相关结果
3. **LLM 查询转换（推荐）** — 将整个对话（含后续问题）传给 LLM，生成一个独立的搜索查询

```
对话历史:
  用户: "意大利有什么好玩的"
  助手: "罗马斗兽场、威尼斯..."
  用户: "那里有什么食物"
LLM 生成独立查询: "意大利有哪些特色美食"
```

#### Multi Query Retrieval（多查询检索）

利用 LLM **生成多个搜索查询**，并行执行检索，然后将所有检索结果合并传入。

适用于复合问题，一个问题可能依赖多个子问题：

```
原始问题: "红袜队和爱国者队谁最近赢得了冠军"
生成子查询:
  - "红袜队上一次赢得冠军是什么时候"
  - "爱国者队上一次赢得冠军是什么时候"
→ 并行检索，合并结果
```

#### RAG-Fusion（RAG 融合）

建立在多查询检索思想之上，但**不再将所有文档直接传入**，而是使用**互惠排名融合（Reciprocal Rank Fusion, RRF）**对多路检索结果进行重新排序，提升检索结果质量。

RRF 的核心公式：每个文档在每路查询中的排名取倒数（`1 / (k + rank)`），然后对所有查询的得分求和，得到最终排序分数。

```
原始问题: "如何优化 RAG 系统性能"
生成多个查询:
  - "RAG 检索优化方法"
  - "提升 RAG 响应质量的技巧"
  - "RAG 系统最佳实践"
→ 并行检索 → RRF 重排序 → 合并结果
```

#### 方法对比

| 方法 | 核心机制 | 适用场景 |
|------|---------|---------|
| Rewrite-Retrieve-Read | LLM 重写查询 | 用户提问含糊或措辞不当 |
| Step Back Prompting | 生成后退问题 + 双重检索 | 需要更广泛的背景知识 |
| Follow Up Questions | 对话压缩为独立查询 | 多轮对话中的上下文保持 |
| Multi Query Retrieval | 生成多个子查询并行检索 | 复合问题需拆分 |
| RAG-Fusion | 多查询 + RRF 重排序 | 追求更高检索质量 |


### RAG 路由

当系统存在多个不同的知识源（如 Python 文档库、JS 文档库、内部 Wiki）时，需要通过路由机制将查询智能分发到最相关的数据源。路由结果可能是 0 个、1 个或多个数据源，多路结果再通过融合策略合并。

#### 逻辑路由

通过 LLM 的结构化输出，将用户查询分类到预定义的数据源。LLM 分析查询意图后，从候选数据源中选择最匹配的一个或多个。

```
用户问题: "Python 的装饰器怎么用"
LLM 路由决策 → datasource: "python_docs"
→ 仅在 Python 文档库中检索

用户问题: "React 和 Vue 哪个更适合大型项目"
LLM 路由决策 → datasource: ["js_docs", "js_docs"]
→ 在两个文档库中分别检索，融合结果
```

优势是可预测、易调试；劣势是灵活性有限，需要预先定义所有候选数据源。

#### 语义路由

基于嵌入向量计算用户查询与各路由描述之间的余弦相似度，选择相似度最高的路由。不依赖 LLM 调用，仅使用嵌入模型，速度更快、成本更低。

```
路由描述:
  - "与物理学、力学、运动相关的问题"
  - "与数学、计算、公式相关的问题"

用户问题: "什么是牛顿第二定律"
→ 计算查询嵌入与各路由描述的余弦相似度
→ 匹配到物理学路由 → 在物理知识库中检索
```

#### 元数据路由

不依赖查询内容本身，而是根据请求附带的上下文元数据（用户角色、商品属性、地域、时间等）决定路由目标。这些信息通常来自用户画像或业务系统，而非 LLM 推理。

```
用户 A (role: "客服") 询问: "退货流程是什么"
→ 路由到客服知识库

用户 B (role: "买家") 询问: "退货流程是什么"
→ 路由到买家帮助文档

用户 C (product_category: "电子产品") 询问: "保修政策"
→ 路由到电子产品售后文档库

用户 D (product_category: "服装") 询问: "保修政策"
→ 路由到服装售后文档库
```

同一个问题因元数据不同被路由到不同知识源，返回与用户身份或业务上下文匹配的答案。

#### 混合路由

结合多种策略：优先使用语义路由（快速、低成本），当相似度低于阈值时回退到逻辑路由（LLM 判断更准确），同时叠加元数据路由作为硬性约束，兼顾效率与准确性。

#### 路由方式对比

| 方式 | 决策依据 | 速度 | 成本 | 可解释性 |
|------|---------|------|------|---------|
| 逻辑路由 | LLM 结构化输出 | 快 | 较高（需 LLM 调用） | 高 |
| 语义路由 | 嵌入相似度 | 很快 | 低（仅需嵌入） | 中 |
| 元数据路由 | 请求上下文（角色/属性等） | 极快 | 无 | 极高 |
| 混合路由 | 多策略组合 | 中 | 中 | 中 |

> 参考：[LangChain - 路由](https://python.langchain.ac.cn/docs/how_to/routing/), [Semantic Router](https://github.com/aurelio-labs/semantic-router)

### 切分策略

Chunking 的核心矛盾：chunk 太小语义不完整，chunk 太大信噪比太低。没有万能的切分策略，只有场景适配。

#### 固定长度切分

按字符数或 Token 数硬切，相邻块之间保留一定重叠（Overlap），例如每 512 Token 切一块，重叠 64 Token。

实现极简、行为可预测，但完全不管语义边界，可能把一句话从中间切断、把完整论述拆成两半。仅适合快速 MVP 验证，不适合生产环境。

#### 递归字符切分

模拟人类阅读方式，按优先级逐层级尝试不同分隔符：先按段落分隔符（`\n\n`）拆 → 仍太大则按换行符（`\n`）拆 → 仍太大则按句号拆 → 依此类推，直到每块都在目标大小以内。

```
原文:
  第一段：介绍 RAG 的背景...\n\n
  第二段：RAG 的核心流程...\n\n
  第三段：存在的问题...

递归切分（目标 200 Token）:
  Chunk 1: "第一段：介绍 RAG 的背景..."（整段保留）
  Chunk 2: "第二段：RAG 的核心流程..."（整段保留）
  Chunk 3: "第三段：存在的问题..."（整段保留）
```

更倾向于在自然语义边界处断开，同时能控制 chunk 上限大小。是大多数普通文本场景下的默认选择，工程实践中最常用的基础策略（典型实现：LangChain 的 `RecursiveCharacterTextSplitter`）。但仍基于字符规则，不理解真正的语义。

#### 语义切分

不按字符或格式切，而是用 Embedding 模型计算相邻句子的语义相似度，当相似度出现明显跌落时判定发生了话题转换，在该处切开。

```
句子序列: [S1, S2, S3, S4, S5, S6]
相似度:    [0.9, 0.85, 0.3, 0.88, 0.91]
                              ↑ 跌落
切分结果:
  Chunk 1: [S1, S2, S3]  ← 话题 A
  Chunk 2: [S4, S5, S6]  ← 话题 B
```

切分结果的语义完整性最好，每个 chunk 内部主题聚焦。但需要对全文做 Embedding 计算，处理速度慢、成本高；且切出的 chunk 大小很不规则，容易产生超小块（实测平均片段仅 43 Token），必须设置合理的 `min_chunk_size`（建议 200-400 Token）。

#### 结构感知切分

利用文档本身的层级结构（Markdown 标题、法律条款编号、代码函数边界、HTML 标签等）作为切分边界。NVIDIA 测试数据显示，Page-Level Chunking 在金融报告和法律文档上平均准确率 0.648，方差最低。

| 文档类型 | 推荐切分方式 | 实现工具 |
|----------|-------------|---------|
| Markdown | 按标题层级（H1/H2/H3）切 | `MarkdownHeaderTextSplitter` |
| HTML | 按标签层级切（h1~h6、p、div） | `HTMLHeaderTextSplitter` |
| PDF | 按页或章节切 | `chunk_by_title`、`chunk_by_page` |
| 代码 | 按函数、类切 | `PythonCodeTextSplitter` |

每个 chunk 天然对应一个有意义的主题，检索精准度最高，效果最稳定。但依赖文档有清晰的结构化标记，非结构化文本不适用。

#### 父子块策略（Parent-Child Chunk）

解决"小块召回准但上下文残缺，大块保留完整但召回噪声大"的矛盾：

1. 先切成约 300 Token 的小块（子块）用于向量检索
2. 每个子块挂载到约 1200 Token 的父段落上
3. 检索时先命中小块，再把对应父段落放入上下文

```
父块（~1200 Token）: "RAG 检索优化完整章节..."
  ├─ 子块 1（~300 Token）: "RAG 检索优化 - 向量检索部分"   ← 索引 & 检索
  ├─ 子块 2（~300 Token）: "RAG 检索优化 - 重排序部分"     ← 索引 & 检索
  └─ 子块 3（~300 Token）: "RAG 检索优化 - 混合检索部分"   ← 索引 & 检索

用户查询命中子块 2 → 返回父块（完整章节上下文）
```

兼顾召回精度和上下文完整性，在长文档、政策解读、故障手册等场景效果明显。代价是索引存储量增加，检索时多一次关联查询。

#### 重叠控制

在相邻 chunk 之间设置重叠区域（Overlap），确保边界处的语义不被硬切断。例如 chunk 大小 512 Token，重叠 64 Token，则相邻两个 chunk 有 64 Token 的重复内容，边界附近的信息不管命中哪个 chunk 都能拿到完整上下文。

重叠太小起不到保护作用，重叠太大则重复内容多、浪费向量空间、增加检索噪声。经验参考值：

| 文档类型 | 建议 Chunk 大小 | 建议重叠 |
|---------|----------------|---------|
| 通用文本 | 512 Token | 50-100 Token |
| 对话型问答 | 256-512 Token | 30-50 Token |
| 知识库检索 | 512-1024 Token | 50-100 Token |
| 法规合同 | 按条、款、项结构切 | — |
| 表格密集文档 | 表格单独作为一块，绝不能跨块切分 | — |

#### 元数据注入/过滤

在文档切分后，为每个分块注入元数据（如文档名称、所属分类、标签、作者、时间等），查询时可根据元数据进行过滤，缩小检索范围，提升检索精度。

```
文档: "Python 3.12 新特性.md"
切分后为每个 chunk 注入:
  metadata:
    doc_name: "Python 3.12 新特性"
    category: "python"
    tags: ["python", "release", "changelog"]

用户查询: "Python 最新版本的改进"
→ 先按 category=python 过滤，再执行语义检索
```

> **递归字符切分 vs 结构感知切分**：表面看都在用"换行、段落、标题"做切分，但驱动力完全不同。递归字符切分是**大小驱动**——它有一个目标 chunk 上限，按优先级递减的分隔符（`\n\n` → `\n` → `。` → ` `）逐级尝试拆分，目的是让每块不超限。它看到的是字符流，`\n\n` 只是一个"碰巧大概率对应段落"的字符模式，并不区分这个换行是在两个段落之间还是标题后面。结构感知切分是**语义层级驱动**——它先解析文档的结构树（Markdown 的 `#`/`##`/`###` 层级、HTML 标签树、法律"条/款/项"编号），然后按层级节点切分，每个 chunk 带着完整的层级路径元数据（如 `{"Header 1": "RAG 概述", "Header 2": "检索策略"}`）。前者"以大小为目标、碰巧利用自然分隔符"，后者"以文档语义层级为目标、显式解析结构树"。

#### 策略对比

| 策略 | 核心机制 | 实现难度 | 成本 | 适用场景 |
|------|---------|---------|------|---------|
| 固定长度 | 按 Token 数硬切 | 低 | 低 | MVP 快速验证 |
| 递归字符 | 多层级分隔符递归切分 | 低 | 低 | 通用文本（默认选择） |
| 语义切分 | Embedding 相似度跌落检测 | 高 | 高 | 质量要求极高、话题切换频繁 |
| 结构感知 | 按文档层级结构切分 | 低 | 低 | 有清晰结构的文档（最优） |
| 父子块 | 小块检索 + 大块上下文 | 中 | 中 | 长文档、手册、法规 |
| 重叠控制 | 相邻 chunk 保留重叠 | 低 | 低 | 所有策略的补充 |
| 元数据过滤 | 注入元数据 + 查询过滤 | 低 | 低 | 有明确分类体系的文档 |

> 参考：[JavaGuide - RAG 文档处理与切分策略](https://javaguide.cn/ai/rag/rag-document-processing.html), [Adaptive Chunking (arxiv)](https://arxiv.org/pdf/2603.25333)

### RAG 检索评估

> 来源：[A Practical Guide to RAG Pipeline Evaluation (Part 1: Retrieval)](https://medium.com/relari/a-practical-guide-to-rag-pipeline-evaluation-part-1-27a472b09893) — Yi Zhang & Pasquale Antonante, Relari.ai

**核心观点**：盲目试技术、凭直觉调参不如建立稳健的评估管道。系统性评估可以加速开发周期，仅凭肉眼观察不应是唯一的评估策略。

#### 检索评估指标

**排名无关指标**（Rank-agnostic Metrics）：

- **Precision（精确率）** — 检索到的上下文中有多少比例是相关的，衡量信噪比
- **Recall（召回率）** — 所有相关上下文中有多少被检索到，衡量完整性
- **F1** — 精确率和召回率的调和平均值

> **Recall 是检索的北极星指标**。只有当检索系统确信检索到的上下文足够完整时，才能保证回答的质量。

**排名相关指标**（Rank-aware Metrics）：

- **MRR（Mean Reciprocal Rank）** — 衡量第一个相关块的平均出现位置，适用于单个块即可回答的场景

  对每个查询，找到第一个相关块的排名 rank，取倒数 1/rank，对所有查询求平均：
  ```
  查询 1: [不相关, 相关✓, 不相关, 相关✓] → 第一个相关块排第 2 → RR = 1/2 = 0.5
  查询 2: [相关✓, 不相关, 不相关, 不相关] → 第一个相关块排第 1 → RR = 1/1 = 1.0
  查询 3: [不相关, 不相关, 相关✓, 不相关] → 第一个相关块排第 3 → RR = 1/3 ≈ 0.33
  MRR = (0.5 + 1.0 + 0.33) / 3 ≈ 0.61
  ```

  只关心"第一个命中的有多靠前"，后面的排名完全忽略。

- **MAP（Mean Average Precision）** — 捕获所有检索到的相关块并计算加权分数，适用于需要综合多块信息的场景

  在排名列表的每个相关结果处，计算"到当前位置为止的 Precision"，对所有相关结果取平均得到 AP，再对所有查询取平均得到 MAP。AP 的分母是相关结果总数（不是检索返回数），遗漏相关结果会自然降低 AP：
  ```
  查询 1: [不相关, 相关✓, 不相关, 相关✓]
    位置2命中 → P@2 = 1/2 = 0.5    位置4命中 → P@4 = 2/4 = 0.5
    AP₁ = (0.5 + 0.5) / 2 = 0.5   （相关结果共2个）

  查询 2: [相关✓, 不相关, 相关✓, 不相关]
    位置1命中 → P@1 = 1/1 = 1.0    位置3命中 → P@3 = 2/3 ≈ 0.67
    AP₂ = (1.0 + 0.67) / 2 ≈ 0.83

  查询 3: [不相关, 不相关, 相关✓, 相关✓]
    位置3命中 → P@3 = 1/3 ≈ 0.33   位置4命中 → P@4 = 2/4 = 0.5
    AP₃ = (0.33 + 0.5) / 2 ≈ 0.42

  MAP = (0.5 + 0.83 + 0.42) / 3 ≈ 0.58
  ```

  MRR 只看第一个命中位置，MAP 看所有命中位置的累积精确率。MRR 适合"只需一条答案"，MAP 适合"需要召回全部相关块"。

- **NDCG（Normalized Discounted Cumulative Gain）** — 考虑相关性分类为非二进制的情况，需要更细粒度加权时使用

#### LLM 作为评估器

在 HotpotQA 数据集上的实验结论：

- GPT-4 对上下文相关性的二元分类准确率仅 **79%**，所有模型倾向过于保守（高精确率、低召回率）
- 多块评估时误分类会累积，**Context Precision/AP 不够可靠**
- GPT-4 的 Context Coverage 在预测 Context Recall 方面准确率 **>80%**，但不能完全替代 Recall —— LLM 评估器无法感知检索上下文之外遗漏了什么

**结论**：LLM 代理指标适合方向性洞察，但不适合精确评估。人工标注的 ground truth 数据集仍然是最可靠的方法。

#### 三条可操作建议

**1. 先追 Recall，再优化 Precision**

如果检索不到正确信息，高精确率也无法将质量提升到生产级别。应根据用例设置 Recall 阈值，越过阈值后再关注 Precision。Precision 仅在 LLM 无法从噪声中分离信号时才成为瓶颈，提高 Precision 可以降低成本和延迟。

```
管道 A: Recall=85%, Precision=60%  ← 先确保 Recall 足够高
管道 B: Recall=75%, Precision=80%
管道 C: Recall=50%, Precision=90%  ← 高 Precision 但 Recall 不足，无法生产使用
```

**2. 用 @K 选择检索多少块**

Recall 和 Precision 之间存在权衡。检索语料库中的所有内容，Recall 保证 100%，但 Precision 极低。通过查看前 K 个检索块的 Precision@K / Recall@K，可以找到合适的 K 值。

```
K=1:  Recall=30%, Precision=95%
K=3:  Recall=65%, Precision=80%
K=5:  Recall=82%, Precision=60%  ← 如果目标是 Recall>80%，K=5 即可
K=10: Recall=85%, Precision=35%  ← 继续增大 K 收益递减
```

**3. 排序质量不足时引入重排序**

当 Recall 阈值需要极大的 K 才能达到时，说明粗检索的排序质量不足，应引入重排序。先用大 K（如 100 个）粗召回，再通过重排序器精排到少量高质量结果（如 5 个），技术原理见下文[重排序（Rerank）](#重排序rerank)小节。

### 重排序（Rerank）

Rerank 的核心思路是 **"Retrieve then Rerank"（检索-重排）两阶段架构**：

**第一阶段用 Bi-Encoder 做粗召回**：Query 和 Document 分别独立编码为向量，通过余弦相似度快速从大规模语料中检索出 Top-K 候选。速度快（文档向量可预计算），但精度有限 —— 编码时 Query 和 Document 没有任何交互，只在最后的向量空间做比较，损失了细粒度语义匹配能力。

**第二阶段用 Cross-Encoder 做精排**：将 Query 和每个候选 Document 拼接后联合输入 Transformer，通过交叉注意力（Cross-Attention）做深层语义交互，为每个文档计算精确的相关性分数，按分数重排序。精度远高于 Bi-Encoder，但需要逐对计算，只适合对少量候选（几十到几百个）做精排。

```
Query + Doc_1 → Cross-Encoder → score_1
Query + Doc_2 → Cross-Encoder → score_2
...
→ 按 score 重排序 → 返回最相关的 Top-N
```

**为什么需要两阶段**：Cross-Encoder 的计算复杂度是 O(Q × D)，对全量文档逐一配对代价不可接受。Bi-Encoder 可预计算所有文档向量，检索是毫秒级。所以用 Bi-Encoder 先缩小候选集（O(D) → O(K)），再用 Cross-Encoder 精排（O(K)），兼顾效率与精度。

**常用 Rerank 模型**：Cohere Rerank（API 服务，开箱即用）、BGE-Reranker（开源，中文效果优秀）、bce-reranker（开源）等，原理均为 Cross-Encoder。

**重排效果评估**：用前文定义的排名相关指标度量重排前后的质量变化——MRR 适合"单个块即可回答"的场景，MAP 适合"需综合多块"的场景，NDCG 适合相关性非二分类的细粒度加权。此外，呈现给 LLM 的块顺序本身也很重要，"Lost in the Middle"论文表明 LLM 对长提示中间位置的信息关注程度衰减，重排序的价值不仅在于筛选，还在于把最相关的内容排到 LLM 关注度最高的位置。

### RAG 评估（生成阶段）

> 来源：[A Practical Guide to RAG Pipeline Evaluation (Part 2: Generation)](https://blog.nuvi.dev/a-practical-guide-to-rag-evaluation-part-2-generation-c79b1bde0f5d) — Yi Zhang & Pasquale Antonante, Relari.ai

评估"生成"阶段可以回答以下关键问题：是否必须使用 GPT-4，还是更小的模型也能胜任？是否应该微调 LLM？哪些提示词能最大程度减少幻觉？答案对不同提示词有多敏感？如果提供了正确的上下文，LLM 是否已经足够好，是否应该把精力集中在改进检索上？

#### 符号定义

- **Q**: 问题
- **C**: 检索到的上下文
- **C***: 真实基准上下文
- **A**: 生成的答案
- **A***: 真实基准答案
- **X ~ Y**: 基于 Y 来评估 X

#### 评估维度

**有基准答案时 — 整体评估（A~A*）**：

- **正确性**：将生成的答案 (A) 与基准答案 (A*) 比较，能从整体上捕捉答案的所有预期特质（基于上下文、相关问题、正确推理、理想风格等）

**无基准答案时 — 多维度评估**：

- **忠实度 / Grounded（A~C）**：LLM 是否使用了检索到的上下文来回答？如果上下文不足以回答问题，LLM 应能识别并回答"我不知道"
- **相关性（A~Q）**：答案是否直接回应了问题，避免重复上下文却不回答问题
- **逻辑性（A~Q, C）**：LLM 是否能根据问题和上下文给出符合逻辑推导的正确答案
- **风格（A~Q 或单独的 A）**：答案是否太长太短、语气是否合适、是否引用了证据

#### 评估指标

**1. 确定性指标**

ROUGE、Token Overlap、BLEU 等。核心思路是计算生成答案与参考文本之间的词汇重叠率。以正确性评估（A~A*）为例，比较生成答案 A 和基准答案 A* 的 Token 重叠比例；以忠实度评估（A~C）为例，比较答案 A 和检索上下文 C 的 Token 重叠比例。计算速度极快（毫秒级，无需 GPU），但根本局限在于无法理解语义——"猫坐在垫子上"和"小猫卧于地毯之上"几乎零重叠，但语义相同。

**2. 语义指标**

使用专用小型语言模型（如 DeBERTa-v3 NLI）做自然语言推理（NLI）。给定句子对（S1, S2），模型输出三个概率：矛盾（S1 与 S2 冲突）、蕴含（S1 暗示 S2）、中立（无确定逻辑关系）。在正确性评估中，可检查生成答案（A）是否蕴含基准答案（A*）的关键信息；在忠实度评估中，可检查答案（A）是否与上下文（C）矛盾。相比确定性指标，语义指标能捕捉同义改写和逻辑推导，但仍然无法处理复杂推理和多步逻辑。

**3. 基于 LLM 的指标**

最通用，理论上可以评估答案的任何方面。关键在于提供正确的提示词，并使用与生成答案不同的、能力足够强的模型作为评估器。

#### 与人工标注的一致性

评估指标是否有实用价值，取决于它能否替代人工标注。验证方法是：让自动指标（确定性/语义/LLM 指标）对每个答案打分，同时让人工标注员对同样的答案打分，然后用统计相关性系数衡量这两组分数的一致程度。作者使用论文《Evaluating Correctness and Faithfulness of Instruction-Following Models for Question Answering》发布的人工标注数据集，包含 600 个忠实度标注和 1200 个正确性标注。三种相关性系数各有侧重：

- **皮尔逊相关系数（Pearson）**：衡量两个变量的线性相关程度（如身高与体重），反映的是数值层面的同步变化
- **斯皮尔曼秩相关（Spearman）**：基于排名衡量单调关系（如两份"最喜欢的运动队"排名列表的一致性），对异常值更鲁棒
- **肯德尔 Tau（Kendall's Tau）**：比较两组排名中一致对与不一致对的比例（如两份"Top 10 电影"列表的排序相似度），适合小样本

**正确性维度（A~A*）**：LLM 指标与人工标注的相关性最高，其中 GPT-4 在三种系数上均排名第一。但值得注意，表现最好的确定性指标和语义指标可以媲美除 GPT-4 之外的其他 LLM 指标（如 GPT-3.5）。这意味着如果不使用 GPT-4 级别的模型，传统指标在正确性评估上并不逊色。

**忠实度维度（A~C）**：结果更加出人意料——简单的确定性指标（如 Token Overlap）与 LLM 指标的差距很小。LLM 指标的表现高度依赖提示词策略：作者测试了两种策略，`binary_prompt`（直接判断答案是否基于上下文，输出二分类）在使用能力较弱的模型时表现显著下降；`by_statement_prompt`（将答案拆解为多个独立陈述，逐一评估每个陈述是否可归因于上下文）才能让 LLM 指标发挥优势。这表明忠实度评估的关键瓶颈不在模型能力，而在于评估粒度——逐句判定比整体判定更有效。

**小结**：LLM 指标在统计上优于传统指标，但优势并不是压倒性的。简单指标同样具备有效性，且在忠实度评估中表现尤其出色。这一发现为降低评估成本提供了理论基础——用简单指标处理大部分数据，只在必要时调用 LLM。

#### 降低评估成本：混合评估流水线

纯 LLM 评估成本高且速度慢，作者提出集成指标 + 混合评估流水线，可将评估成本降低 15 倍且不降低质量：

**确定性-语义集成**表现最佳，平衡了精确率和召回率，正确性分类的精确度甚至超过所有单一指标。

**混合流水线步骤**：

1. 用确定性-语义集成对所有数据点进行首轮评估
2. 用**保形预测（Conformal Prediction）**量化预测置信度（如"该答案正确的概率至少为 90%"），将不确定的数据标记为"未决"
3. 仅对约 7% 的"未决"数据点使用 GPT-4 做最终判定

结果：匹配了纯 GPT-4 评估器的性能，但由于只在 7% 的数据上使用了 GPT-4，成本降低了约 15 倍。

### 嵌入（Embedding）

#### 嵌入模型选择

嵌入模型负责把文本转成高维向量，不同模型在维度、性能、成本、多语言支持上差异很大：

- **OpenAI**（`text-embedding-3-small/large`）：通用性强，API 调用，按 token 计费，适合快速起步
- **BGE**（北京智源）：开源，有中文特化版本（`bge-large-zh`），可本地部署，MTEB 排名靠前
- **Cohere**（`embed-v3`）：多语言表现好，支持不同任务类型（search_document / search_query）
- 其他还有 **E5**、**GTE**、**Jina** 等

选择的核心考量：是否需要本地部署、中文效果、延迟要求、成本预算。

#### 微调嵌入模型以适配领域

通用嵌入模型在垂直领域（医疗、法律、金融）效果可能打折，因为术语的语义距离在领域内有不同含义。做法是在领域数据上继续训练嵌入模型，让同义文本的向量更近、无关文本更远。常用方式：

- 准备领域内"查询-文档"正负样本对
- 用对比学习（Contrastive Loss）微调
- 代表框架：`sentence-transformers`、`FlagEmbedding` 的 fine-tune 工具

关键判断标准：通用模型在你的领域数据集上的 recall@k 如果已经够用，就不需要微调。

#### 稀疏嵌入/混合检索（BM25 + Dense Retrieval）

这是两种不同的检索范式组合，需要先理解稀疏检索的核心概念。

**稀疏检索与 BM25**：

- **反向索引（Inverted Index）** 是底层数据结构——记录"哪个词出现在哪些文档中、出现几次、在什么位置"。几乎所有文本检索系统（包括 Elasticsearch）的底层都靠它
- **BM25** 是构建在反向索引之上的**排序算法**，基于 TF-IDF 的改进，公式核心考虑三个因素：
  - **TF（词频）**：词在文档中出现越多，越相关，但有饱和上限（不会线性增长）
  - **IDF（逆文档频率）**：越稀有的词权重越高，常见停用词权重趋近于零
  - **文档长度归一化**：长文档不会因为词多就占便宜
- **稀疏检索（Sparse Retrieval）** 是一个更宽泛的范式名称，叫"稀疏"是因为每个文档/查询的向量表示是超高维（词表大小，通常几万到几十万维）但极其稀疏（绝大部分维度是 0，只有出现过的词才有值）。BM25 是其中最经典的代表，此外还有 SPLADE 等学习型稀疏方法

三者的关系：稀疏检索（范式） → BM25（排序算法） → 反向索引（底层数据结构）

**混合检索**：

- **稀疏检索（BM25）**：基于词频的"精确匹配"，查询里出现"Transformer 架构"，只有包含这些词的文档才能命中。速度快、可解释性强，但无法理解语义
- **稠密检索（Dense Retrieval）**：用嵌入模型做向量相似度搜索，能捕捉语义关系（"深度学习模型" ≈ "神经网络"），但可能遗漏精确关键词匹配

混合检索把两者结合：BM25 负责捞关键词精准匹配的，向量检索负责捞语义相关的，两路结果通过加权融合（如 Reciprocal Rank Fusion, RRF）合并后排序。实践中混合检索几乎总是优于单独任何一种。

总结：**选模型决定基线，微调提升领域适配，混合检索补齐精确匹配短板**——三者是递进关系，不是三选一。

### 索引构建

检索的前提是数据已经被组织成可检索的结构。索引构建决定"数据怎么存、怎么找"，是 RAG 流水线中承上启下的关键环节。常见的三种索引方式各有侧重：

#### 向量索引

把文档切块后用嵌入模型转成高维向量，存进专门的向量数据库。检索时把用户 query 也转成向量，做近似最近邻搜索（ANN），找到语义最相似的文档块。这是 RAG 最基础的索引方式，对应前文"稠密检索"。

常见的向量数据库/检索库在定位上有差异：

- **FAISS**（Meta 开源）：纯算法库，不是数据库。性能极高，适合单机、离线场景，但不支持增删改、持久化、分布式
- **Milvus**：分布式向量数据库，支持海量数据、实时增删、云端部署，适合生产级系统
- **Pinecone**：全托管 SaaS 向量数据库，零运维，按用量付费，适合快速上线但数据需出域
- **Weaviate**：支持向量 + 关键词混合检索，内置多种嵌入模型集成，GraphQL API
- **Qdrant**：Rust 实现的高性能向量数据库，支持过滤、分片，轻量且性能好
- **Chroma**：轻量级嵌入式向量数据库，适合原型开发和中小规模场景

选择核心考量：数据规模、是否需要分布式、是否接受数据出域、运维成本、是否需要混合检索能力。

#### 层级索引（Summary + Detail）

解决的问题是：文档切块后每个 chunk 只有局部信息，检索时容易丢失全局上下文，且在大规模文档库上直接搜全量 chunk 噪音太多。

做法是把文档组织成两层：

- **Summary 层**：对整个文档（或大段落）生成摘要，建索引
- **Detail 层**：对文档的原始 chunk 建索引

检索时分两步走：先在 Summary 层找到相关文档，再到该文档的 Detail 层找具体 chunk。类似"先找书，再翻页"。在数万篇以上的文档库中特别有效。

这和前文的"父子块策略"思路类似但粒度不同——父子块是 chunk 级别的局部-全局关系，层级索引是文档级别的粗-细两级检索结构。

#### 知识图谱索引（Graph RAG）

把文档中的实体和关系抽取出来构建知识图谱（节点 = 实体，边 = 关系）。检索时不是搜"哪段文本相似"，而是沿着图谱关系做推理和遍历。

核心优势是能回答需要**多跳推理**的问题。比如问"A 公司的母公司的 CEO 还担任哪些公司的董事"，向量检索很难一次命中，但图上可以沿着 `A公司 → 母公司 → CEO → 担任董事的其他公司` 这条路径走通。

微软的 **GraphRAG** 是目前最有代表性的实现：先用 LLM 从文档中抽取实体和关系建图，查询时通过社区检测算法把图谱分成不同层级的社区摘要，然后结合全局搜索和局部搜索来回答问题。

代价也很明显：建图阶段需要大量 LLM 调用来做实体/关系抽取，成本高、延迟大，适合知识密集型且需要复杂推理的场景。

#### 三种索引方式对比

- **向量索引**解决"语义相似"的基础检索问题，是所有 RAG 系统的默认选择
- **层级索引**解决大规模文档的"先粗后细"检索效率问题，在文档量大时几乎是必选项
- **知识图谱索引**解决"多跳推理"的复杂关联查询问题，代价高但能力天花板也高

三者可以组合使用（比如向量索引 + 层级索引组合，或在向量索引基础上叠加知识图谱做增强），不是互斥的。

### 生成阶段优化

检索阶段负责"找到相关内容"，生成阶段负责"基于检索到的内容产出答案"。即使检索质量很高，如果生成阶段没有做好约束，仍可能出现幻觉、答非所问、无法溯源等问题。以下是四个关键优化方向。

#### 提示词工程（Prompt Engineering for RAG）

RAG 生成阶段的提示词设计直接决定回答质量。核心目标：**让 LLM 只用检索到的内容回答，不编造信息**。

**基本约束模式**：

```
System: 根据以下检索到的文档回答用户问题。
仅使用提供的文档内容，不要使用先验知识。
如果文档中没有足够信息回答问题，请回答"根据检索到的文档，无法回答该问题"。

检索到的文档:
{documents}

用户问题: {query}
```

关键点：必须明确告诉 LLM "我不知道"是可接受的回答，否则模型倾向于编造答案来"帮"用户。

**提取式回答**：适用于法律、医疗等要求零偏差的场景，直接返回原文片段而非改写：

```
从检索到的文档中提取最能回答该问题的原文段落，不做任何改写或总结。
```

**Few-shot 示例驱动引用**：在提示词中给出带引用的示例，让模型学习引用格式：

```
示例:
文档[1]: 天空在傍晚呈红色，在清晨呈蓝色。
文档[2]: 天空为红色时，空气中的湿度较高。
问题: 什么时候空气湿度较高？
答案: 当天空呈红色时空气湿度较高 [2]，而天空在傍晚呈红色 [1]。
```

**结构化输出控制**：通过 API 参数（如 OpenAI 的 `response_format={"type": "json_object"}`）或提示词约束输出格式，适用于需要程序化消费答案的场景。

**不同提示策略的权衡**：

| 策略 | 忠实度 | 灵活性 | 适用场景 |
|------|--------|--------|---------|
| 基本约束 | 中 | 高 | 通用问答 |
| 提取式回答 | 极高 | 低 | 法律、医疗、合规 |
| Few-shot 引用 | 高 | 中 | 需要溯源的知识问答 |
| 结构化输出 | 中 | 低 | 程序化消费（API 调用） |

#### 引用溯源（Citation/Attribution）

引用溯源让 RAG 系统的每个回答都能追溯到具体来源，是建立用户信任的关键机制。

**论文"Correctness is not Faithfulness in RAG Attributions"(Wallat et al., 2024) 区分了两个维度**：

- **引用正确性** — 被引用的文档是否实际支持该陈述？
- **引用忠实性** — 引用是否真实反映了模型生成回答时所依据的来源？

即使引用"碰巧"正确，若模型实际未参考该来源（而是使用了内部知识），仍属不忠实引用。

**实现方法**：

**行内引用**是最常用的方式：在提示词中对检索文本预编号 `[1]`, `[2]`，通过指令引导模型在生成时插入引用标记。实现简单，但引用质量依赖 LLM 的遵从能力。

```
文档[1] Python 官方文档: "列表推导式是 Python 中创建列表的简洁方式..."
文档[2] Stack Overflow: "列表推导式在处理大数据集时可能比 for 循环慢..."

问题: Python 列表推导式的性能如何？
答案: 列表推导式是创建列表的简洁方式 [1]，但在大数据集上性能可能不如传统 for 循环 [2]。
```

**后置归因**先生成无引用答案，再在检索文档中搜索支持证据，最后整合引用。实现分离了生成与归因，灵活性高但可能存在归因不准确。代表方法有 ALCE 框架、AGREE（微调 LLM 自动引用检索文档片段）。

**高亮归因**通过前端交互将输出部分与来源做视觉关联（如颜色标记、工具提示），粒度最细但实现成本高，通常需要配合前端开发。

**聚合来源列表**在答案末尾附加 "参考来源" 列表，实现最简单但粒度最粗——无法区分哪个事实对应哪个来源。

| 方法 | 粒度 | 实现复杂度 | 准确性 |
|------|------|-----------|--------|
| 行内引用 | 句级 | 低 | 中（依赖 LLM 遵从能力） |
| 后置归因 | 句级 | 中 | 中（生成与归因分离） |
| 高亮归因 | 词/句级 | 高 | 高 |
| 聚合来源列表 | 文档级 | 极低 | 低 |

LlamaIndex 的 `CitationQueryEngine` 提供了开箱即用的行内引用实现，会将检索到的大块进一步分割成更小的、可独立引用的片段。

#### 幻觉检测与缓解

RAG 场景中的幻觉分为两类：

- **内在幻觉** — 生成内容与检索上下文**相矛盾**（上下文说 A，模型说非 A）
- **外在幻觉** — 生成内容**无法从检索上下文验证**，但可能在外部知识中正确（模型使用了自身参数知识而非检索内容）

**成因**：检索阶段的问题（低质量数据、检索粒度不当、"中间位置诅咒"——Transformer 对长文本中间位置信息注意力衰减）和生成阶段的问题（上下文噪声干扰、预训练知识与检索上下文冲突、谄媚效应）都会导致幻觉。

**检测方法**：

**NLI（自然语言推理）模型**：用专门的 NLI 模型（如 DeBERTa-v3）判断生成句子是否被上下文蕴含（entailment）。RAGAS 的 Faithfulness 指标即基于此思路——将回答拆分为独立声明，逐条验证是否可从上下文推断：

```
Faithfulness = 被上下文支持的声明数量 / 回答中声明总数
```

**LLM-as-Judge**：使用独立的 LLM 评估生成回答的忠实度。方法包括自一致性检查（多次采样比较答案一致性）和交叉验证（用不同模型互相验证）。灵活但成本高、自身也可能产生误判。

**专用幻觉检测模型**：Vectara 的 HHEM-2.1-Open 是基于 T5 的开源分类器，专为 RAG 幻觉检测设计，可免费用于生产环境。

**缓解策略**：

| 层级 | 策略 | 原理 |
|------|------|------|
| 检索优化 | Self-RAG（按需检索） | 通过检索标签评估是否需要检索，避免引入不相关上下文 |
| 检索优化 | 迭代检索（IRCoT） | 检索和推理交替进行，逐步缩小信息范围 |
| 上下文降噪 | 重排序 | 用 Cross-Encoder 过滤低相关性候选 |
| 上下文降噪 | 提示压缩（LLMLingua） | 压缩冗余 Token，减少噪声干扰 |
| 知识冲突处理 | 警惕提示 | 系统提示中提醒 LLM 对潜在虚假信息保持警惕 |
| 事后修正 | RARR | 生成后用独立模型检测并修正不一致内容 |

**实践建议**：检测和缓解应组合使用——先用提示词约束（"仅使用上下文回答"）做基础防护，再用 NLI 或 LLM 评估做后置验证，形成多层防线。对于高可靠性场景（医疗、法律），建议叠加人工抽检。

#### 流式输出与 Chain of Thought

**流式输出（Streaming）**：

RAG 的流式输出架构与普通 LLM 不同——检索阶段必须完成后才能开始生成，无法流式化：

```
用户查询 → [检索阶段（同步）] → 获取上下文 → [LLM 生成（流式 SSE）] → 逐 Token 推送
```

主流实现使用 SSE（Server-Sent Events）协议。用户感知流式界面比缓冲响应快约 40%（即使总时间相同），体验提升显著。可以在等待检索时先发送状态事件（如 "正在检索文档..."）来进一步降低感知延迟。

关键监控指标：TTFT（首 Token 时间）衡量用户首次看到内容的时间，TPS（每秒 Token 数）衡量生成速度。

**Chain of Thought（CoT）在 RAG 中的角色**：

CoT 让 LLM 在回答前先展示推理过程（如"根据检索到的三段内容，首先...其次..."），提升复杂问题的回答质量。

但近期研究揭示了 CoT 在 RAG 中的**忠实性问题**：

- Li et al.（ACL 2025）发现模型能够绕过 CoT 中的推理链，直接从问题中提取关键线索——即 CoT 并**没有忠实地反映模型实际的推理过程**
- Tanneru et al.（2024）证明通过提示、微调、修改内部激活等方式引出忠实 CoT 都存在**固有困难**，现有干预手段可能不足以解决

这意味着 CoT 提升的是**可解释性和复杂推理能力**，但不保证推理过程本身忠实于检索上下文。

**实践建议**：

| 场景 | 推荐策略 |
|------|---------|
| 简单事实问答 | 直接约束式提示，不用 CoT |
| 复杂多跳推理 | CoT + RAG 组合（如 CoT-RAG 框架） |
| 法律/医疗等高忠实性要求 | 提取式回答，不用 CoT |
| 需要可解释性 | 使用 CoT，但需注意推理过程可能不完全忠实 |

> 参考：[Prompt Engineering Patterns for RAG - MachineLearningMastery](https://machinelearningmastery.com/prompt-engineering-patterns-successful-rag-implementations/), [Hallucination Mitigation for RAG - Mathematics 2025](https://www.mdpi.com/2227-7390/13/5/856), [CoT-RAG - EMNLP 2025](https://aclanthology.org/2025.findings-emnlp.168/), [Correctness is not Faithfulness in RAG Attributions](https://arxiv.org/abs/2412.18004), [RAGAS Faithfulness](https://docs.ragas.io/en/latest/concepts/metrics/available_metrics/faithfulness/)

### 缓存策略

RAG 管道包含多个计算密集步骤（嵌入、检索、重排序、LLM 推理），每一步都可以独立缓存。工程实践中的核心原则是**多层穿透式缓存**——每层缓存未命中才进入下一层：

```
用户查询 → [语义响应缓存] → [嵌入向量缓存] → [检索结果缓存] → [重排序缓存] → [LLM 推理]
```

#### 语义缓存

语义缓存用**向量相似度搜索替代精确字符串匹配**来判定缓存命中：将查询转为 embedding，在已缓存查询的向量空间中搜索，相似度超过阈值则直接返回缓存的 LLM 响应，跳过整个管道。

核心价值是能识别语义等价但措辞不同的查询：

```
"What foods are known in Madrid?"        ─┐
                                           ├→ 命中同一缓存条目
"What do people usually eat in Madrid?"  ─┘
```

**相似度阈值**是最关键的参数，直接决定命中率与准确性的平衡：

| 阈值（余弦相似度） | 效果 | 适用场景 |
|-------------------|------|---------|
| ≥ 0.98 | 极严格，几乎不命中 | 高精度要求 |
| 0.85-0.90 | 推荐起始值 | 通用 RAG 场景 |
| 0.80 | 宽松，长查询（>50 字符） | 客服 FAQ |

实测数据（来源：Pedro Medinilla, 2025）：缓存命中时延迟从 13.93 秒降至 0.31 秒，**44 倍加速**，API 调用成本降为零。

主要工具实现：

| 工具 | 核心架构 | 特点 |
|------|---------|------|
| **GPTCache**（Zilliz） | Adapter 拦截 → Embedding → 向量检索 → 相似度评估 | 模块化设计，支持多种向量存储和淘汰策略 |
| **RedisSemanticCache**（LangChain） | Redis + RediSearch 向量搜索 | 需要 redis-stack（含 RediSearch 模块） |
| **LlamaIndex** | 多级缓存（内存/磁盘/Redis） | 语义缓存比精确匹配提升 40-60% 命中率 |

#### 精确缓存

精确缓存按**精确键**存储计算结果，不涉及语义判断，零误命中风险：

**嵌入向量缓存**（最高优先级）：用 `sha256(model + text)` 作键，相同文本 + 相同模型必然产出相同向量，可永久缓存。重新摄入数据时，之前见过的文本块**零 API 调用**。

**检索结果缓存**：用 `query + k` 作键，配合 TTL 控制新鲜度：

```
文档频繁变更 → TTL 60s
稳定内容     → TTL 3600s
高稳定内容   → TTL 86400s
```

**缓存失效策略**：

| 策略 | 适用层 | 原理 |
|------|--------|------|
| 内容寻址（CAS） | 嵌入向量、摘要 | `sha256(content)` 作键，内容变即失效 |
| TTL 过期 | 检索结果、LLM 响应 | Redis `setex(key, ttl, value)` |
| 事件驱动失效 | 全部层 | 文档 CRUD 操作时主动清除关联缓存 |
| 版本化缓存键 | 全部层 | key 包含 `doc_id + kb_version + model_version` |

#### 学术前沿：KV Tensor 缓存

北京大学 & 字节跳动的 RAGCache 论文（arXiv:2404.12457）提出直接缓存 LLM 推理中的 KV 张量（而非文本），用前缀树组织中间状态实现跨请求共享。实测 TTFT 降低 1.2-4 倍，吞吐量提升 1.3-2.1 倍。

#### 缓存策略推荐优先级

```
嵌入向量精确缓存（必用，零风险）→ 检索结果缓存（推荐，配合 TTL）→ 重排序缓存
→ 语义响应缓存（按需）→ LLM 响应缓存（谨慎，仅限静态 FAQ 场景）
```

> 参考：[Beyond Prompt Caching: 5 More Things You Should Cache in RAG Pipelines](https://explore.n1n.ai/blog/beyond-prompt-caching-rag-pipeline-optimization-2026-03-21), [GPTCache - DeepWiki](https://deepwiki.com/zilliztech/GPTCache), [How to Save Costs with Semantic Cache](https://www.pedromebo.com/blog/en-llm-semantic-cache), [RAGCache - arXiv:2404.12457](https://ar5iv.labs.arxiv.org/html/2404.12457)

### 数据管道

"垃圾进，垃圾出"——研究表明 **70% 的召回失败源于入库阶段的数据质量问题**。数据管道覆盖从原始文档到可检索索引的完整链路。

#### 文档解析

文档解析是 RAG 的基石，解析不当直接导致下游幻觉。**PDF 解析是最复杂的挑战**——PDF 本质上是一组打印指令的集合，不是结构化数据格式：

| 挑战 | 具体表现 |
|------|---------|
| 表格误解读 | 行列关系丢失，数值错位 |
| 多栏布局 | 左右栏文本交错，阅读顺序被打乱 |
| 扫描文档 | 需要 OCR，引入识别噪声 |
| 格式丢失 | 标题与正文混淆，层级消失 |

主流解析工具按复杂度分三层：

**轻量级（秒级，适合简单文档）**：

| 工具 | 适用场景 | 局限 |
|------|---------|------|
| PyPDF | 纯文本型 PDF | 无法处理表格、图表 |
| pdfplumber | 文本型 PDF，保留坐标信息 | 视觉元素处理有限 |
| MarkItDown（Microsoft） | 数字原生 PDF、Office 格式 | 无 OCR，表格损坏，多栏失败 |

**模型驱动（十秒级，适合复杂文档）**：

| 工具 | 表格能力 | 公式/多栏 | OCR | 特点 |
|------|---------|----------|-----|------|
| **Docling**（IBM） | 优秀 | 良好 | 自动路由 | 258M 参数布局模型，输出完美 Markdown 表格 |
| **Marker** | 优秀 | 保留 LaTeX | 支持 | 三模型架构，推荐 GPU |
| **Unstructured.io** | 良好 | 有限 | 有限 | 生态最完善，支持多种格式 |
| **LlamaParse** | 良好 | 良好 | 有限 | 云服务，输出 LLM 兼容 Markdown |

实测数据（14 页 PDF）：

| 工具 | 速度 | 表格保真度 | 多栏布局 |
|------|------|-----------|---------|
| MarkItDown | 0.6 秒 | 损坏 | 交错 |
| Docling | 41 秒 | 完美 | 正确 |
| Marker | 2 分 14 秒 | 优秀 | 正确 |

**生产推荐架构**：MarkItDown 作为快速路径 → 输出质量差时回退到 Docling → 仅检测到 LaTeX 内容时调用 Marker（GPU 池）。

#### 数据清洗

六层清洗流水线：

```
原始数据源 → 文档解析 → 文本规范化 → 语义分块 → 元数据富化 → 质量评估&去重 → 向量化入库
```

**文本规范化**是必须执行的标准化操作：Unicode 规范化（全角/半角统一）、清除零宽字符和不可见控制字符、统一换行符、压缩连续空白、清除页眉页脚噪声。

**质量评分**：对每个文本块计算多维分数（有效字符比率 × 0.5 + 重复行检测 × 0.3 + 平均句长 × 0.2），低于阈值的 chunk 直接丢弃。

**去重**采用两阶段流程：

```
全量 Chunks
  → 第一阶段：MinHash LSH 快速近似去重（相似度 > 0.9 → 保留最新版本）
  → 第二阶段：向量余弦相似度精确比对（余弦 > 0.95 → 去重）
```

Google 在 C4 语料库上的研究证明：移除近似重复数据可减少逐字抄袭并提高下游质量，同样适用于 RAG 知识库。

**健康指标**：

| 指标 | 健康范围 |
|------|---------|
| 有效 chunk 率 | > 85% |
| 平均 token 数 | 150-300 |
| 重复率 | < 5% |
| 表格覆盖率 | > 90% |

#### 数据更新

知识库不是静态的，需要持续保持索引新鲜度。按内容波动性分类管理：

| 波动性 | 示例 | 更新频率 | 技术 |
|--------|------|---------|------|
| 高（动态） | 新闻、产品库存 | 秒-分钟级 | Webhooks、CDC |
| 中（周期性） | 报表 | 小时-天级 | 批处理、增量扫描 |
| 低（静态） | 政策文档 | 周-月级 | 完整重索引 |

**变更检测**是增量更新的核心——用 SHA-256 哈希判断文档是否变更，仅处理增量部分：

```
文档源 → 变更检测（哈希比对）→ 消息队列 → 解析工作器 → 分块 → 嵌入 → 向量数据库
```

**增量索引 vs 全量重建**：日常更新用增量 Upsert（删除旧块 + 插入新块），仅当模式变更或索引损坏时才全量重建。关键优化：复用未变更文本块的嵌入向量，仅为变更部分计算新嵌入。

### 可观测性

RAG 可观测性 ≠ 普通 API 监控。核心差异在于需要监控**语义质量**——检索是否命中、Query rewriting 是否有效、LLM 是否忠实引用上下文、Embedding 是否发生漂移。

#### Tracing（分布式追踪）

RAG 管道存在至少 5 个**"静默故障"阶段**——不会产生异常或非 200 状态码，但系统输出已严重退化：

| 阶段 | 静默故障 |
|------|---------|
| Embedding | 模型版本变更，相似度分数漂移，检索质量缓慢退化 |
| Vector Search | 返回零结果（索引过时、阈值过严），LLM 以空上下文产生幻觉 |
| Reranking | 所有候选评分低于截止值，结果集从 10 个缩减为 0 个 |
| Context Assembly | 检索块超过上下文窗口，组装步骤静默截断 |
| LLM Generation | `finish_reason = "length"`（达到 max_tokens），截断文本被当作完整答案 |

一个良好仪表化的 RAG 管道在父级 `rag.query` span 内产生 5 个子 span：

```
rag.query (父 Span)
  ├─ rag.query_embedding    → 嵌入模型、耗时
  ├─ rag.retrieval          → top_k、返回结果数、是否空结果
  ├─ rag.reranking          → 重排模型、候选分数列表
  ├─ rag.context_assembly   → token 数、是否截断
  └─ rag.llm_generation     → 模型、token 用量、finish_reason
```

其中 **`rag.retrieval.empty_result`**（检测检索静默失败）和 **`rag.context.truncated`**（检测上下文截断）是驱动生产告警的两个最关键属性。

#### 监控指标

RAG 系统需要三层指标体系：

**系统级**：端到端延迟（p95 > 3s 告警）、检索延迟（p95 > 500ms 告警）、LLM 延迟、QPS、内存占用、索引大小增长。

**检索级**：Recall@K（≥ 80%）、Precision@K、Hit Rate（≥ 95%）、MRR（> 0.7）、空检索率（> 5% 告警）、检索相似度分布。

**语义质量级**：Faithfulness（忠实度，回答是否由检索文档支持）、Answer Relevancy（答案相关性）、Context Precision（上下文精确度）、幻觉率。

**漂移检测**是长期运维的关键：

- **Embedding 漂移**：不同时间写入的 embedding 可能来自不同模型版本，导致相似度下降。建议添加 `embedding_model_version` 字段，监控检索相似度分布变化
- **响应漂移**：自动化的 n-gram、长度或嵌入空间漂移测量，对意外的风格或主题变化发出警报

#### 评估与反馈循环

生产级 RAG 需要三层评估模型：

1. **追踪层** — 了解系统当前行为：捕获输入、输出、计时、token 使用
2. **组件级评估** — 优化单个组件：用 LLM-as-a-Judge 对每个检索 chunk 相关性打分，数据驱动调参
3. **端到端评估** — 衡量用户体验：忠实度、相关性、正确性

评估框架：

| 工具 | 核心能力 |
|------|---------|
| **RAGAS** | faithfulness、answer_relevancy、context_precision/recall，无需 ground truth |
| **DeepEval** | 语义质量评估 + CI/CD 集成，可在 CI 中自动验证 |
| **TruLens** | RAG Triad（上下文相关性、事实性、答案相关性） |

**反馈循环**将生产数据回流到评估集：从生产流量中采样边缘案例 → 加入评估集 → 新版本发布前在评估集上验证 → 回归则阻断合并。这是持续保持 RAG 质量的核心机制。
