### **背景**

![](/雪花算法/log.png)

雪花算法的机器标识是相同的, 导致相同毫秒时生成id冲突.



### **雪花算法**

1. 第一位0, 表示正数
2. 41位时间戳, 因为只有41位, 所以是一个相对起始时间的偏移值, 因此起始时间不能修改
3. 10位机器标识, 分为5位datacenterId, 5位workerId
4. 12位序列号, 同一毫秒内的自增序列号



### **mybatis**

`com.baomidou.mybatisplus.core.toolkit.Sequence`目前使用的是该类生成id

```
protected long getMaxWorkerId(long datacenterId, long maxWorkerId) {
    StringBuilder mpid = new StringBuilder();
    mpid.append(datacenterId);
    String name = ManagementFactory.getRuntimeMXBean().getName();
    if (StringUtils.isNotBlank(name)) {
        /*
         * GET jvmPid
         */
        mpid.append(name.split(StringPool.AT)[0]);
    }
    /*
     * MAC + PID 的 hashcode 获取16个低位
     */
    return (mpid.toString().hashCode() & 0xffff) % (maxWorkerId + 1);
}
```

生成workerId时只用了datacenterId和进程id, 但是在容器中, 进程id都是1, 所以只要datacenterId冲突了, 那么workerId就是冲突的. 等于原来是1/1024的概率冲突, 现在只有1/32的概率冲突.



### hutool

`cn.hutool.core.lang.Snowflake`切换成hutool的雪花算法实现. 阿里云的机器可能会有10ms的时钟回拨, hutool的实现容忍可以配置大于10ms

1. 两者起始时间一致
2. hutool允许的时钟回拨可配置, mybatisplus的固定为5毫秒
3. mybatisplus时钟回拨策略是wait, hutool的是切换到当前缓存时间戳



### **leaf**

[Leaf——美团点评分布式ID生成系统](https://tech.meituan.com/2017/04/21/mt-leaf.html),文档中也介绍了其他一些方案

1. 步长获取
2. 双buffer, 在耗尽前提前获取
3. 高可用, 个人感觉数据库不可用了也不用担心id不能生成了
4. leaf-snowflake, 使用zk保证workerId不冲突, 当然更复杂, 有上报什么的
